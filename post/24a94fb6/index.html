软件测试概述

发表于2022-05-08|更新于2022-05-28|编程
|阅读量:

软件危机

  • 软件生产不能满足日益增长的软件需求
  • 软件生产率随着软件规模和复杂性提高而下降
  • 软件开发的进度与成本失控
  • 软件系统实现的功能与实际需求不符
  • 软件难以维护
  • 软件文档配置没有受到足够的重视

软件过程模型

  1. 瀑布过程模型
  2. 螺旋过程模型
  3. 增量过程模型
  4. 快速原型过程模型
  5. 敏捷过程原型

软件质量：

  • 能满足给定需求的特性。软件需求是衡量软件质量的基础，不符合需求的软件就不具备好的质量。设计的软件应在功能、性能方面等都符合要求，并可靠的运行
  • 具有所期望的各种属性的组合的程度，即软件结构良好，合理地利用系统资源，易读、易于理解，并易于修改，方便软件的维护。
  • 能满足用户综合期望的程度，软件系统具有友好的用户界面，便于用户使用。
  • 软件的组合特性。软件生存周期中各阶段文档齐全、规范，便于配置管理

内部质量

  • 可维护性
  • 灵活性
  • 可移植性
  • 可重用性
  • 可读性
  • 可测试性
  • 可理解性

外部质量

  • 正确性
  • 可用性
  • 效率
  • 可靠性：系统在一定条件下执行特定功能的能力
  • 完整性
  • 适应性
  • 精确性：系统不受错误影响的程度
  • 坚固性：系统在无效输入或压力环境中能继续执行其功能的能力

软件测试

软件测试的目的

  • 测试是一个程序的执行过程，其目的在于发现错误
  • 一个好的测试用例很可能会发现至今尚未察觉的错误
  • 一个成功的测试是发现执行尚未察觉的的错误的测试

软件测试的原则

  1. 尽早测试
  2. 全面测试
  3. 全过程测试
  4. 独立的、迭代的测试
  5. Pareto原则。测试发现的错误中的80%可能起源于20%的模块中
  6. 对测试出的错误结果一定要有一个确认的过程
  7. 制定严格的测试计划
  8. 完全测试是不可能的
  9. 注意回归测试的关联性
  10. 妥善保存一切测试过程文档

软件测试过程模型

V模型

V模型指出：单元测试和集成测试应检测程序的执行是否满足软件设计的要求；系统测试应检测系统功能、性能的质量特性是否达到系统要求的指标；验收测试确定软件的实现是否满足用户需要或合同的要求

局限性： 它仅仅把测试作为在编码之后的一个阶段，是针对程序运行的寻找错误的活动，而忽视了测试活动对需求分析、系统设计等活动的验证和确认的功能。

W模型

W模型强调测试伴随着整个软件开发周期，而且测试的对象不仅仅是程序，需求、设计等同样需要测试，也就是说，测试和开发时同步进行的。

局限性：在W模型中，需求、设计、编码等活动被视为串行的，同时，测试和开发活动也保持着一种线性的前后关系，上一阶段完全结束，才可正式开启下一个阶段工作。这样就无法支持迭代的开发模型。

X模型

X模型的左边描述的是针对单独程序片所进行的相互分离的编码和测试，此后将进行频繁的交接，通过集成最终成为可执行的程序，然后再对这些可执行程序进行测试，此后将进行频繁的交接，通过集成最终成为可执行的程序，然后再对这些可执行程序进行测试。已通过集成测试的产品可以进行封装并提交给用户，也可以作为更大规模和范围内集成的一部分。

  • 可能对测试造成人力、物力和财力的浪费，对测试员的熟练程度要求比较高

H模型

H模型将测试活动完全独立出来，将测试准备活动和测试执行活动清晰地体现出来

H模型揭示了软件测试是一个独立的流程，贯穿产品整个生命周期，与其他流程并发地进行。H模型指出软件测试要尽早准备，尽早执行。

软件测试的分类

按测试方式分类

  • 静态测试：不需要执行所测试的程序，查询代码是否符合规范，对程序的数据流和控制流进行分析。
  • 动态测试：选择实际测试用例运行所测试程序，模拟用户输入。

按测试方法分类

  • 白盒测试
  • 黑盒测试
  • 灰盒测试

按测试过程分类

  • 单元测试（大量采用白盒测试）
  • 集成测试（较多采用黑盒测试）
  • 系统测试
  • 验收测试（通常采用黑盒测试）

按测试目的分类

  • 功能测试

  • 健壮性测试

  • 接口测试

  • 性能测试

  • 强度测试

  • 压力测试

  • 可靠性测试

    安全测试 用户界面测试 安装/反安装测试 文档测试 恢复测试 兼容性测试 负载测试

软件测试流程

  1. 制定测试计划

    1. 软件测试背景（项目介绍、项目人员等）
    2. 软件测试依据（需求文档、软件规格书、软件设计文档等）
    3. 测试范围的界定

  2. 设计测试方案

    标准：是否可以发现尚未发现的软件缺陷？ 是否可以覆盖全部的测试需求？

  3. 测试准备和测试环境的建立

  4. 执行测试

    单元测试 —> 集成测试 —> 系统测试 —> 验收测试，其中每个阶段都包括回归测试

    执行测试的步骤：

    1. 输入
    2. 执行过程
    3. 检查过程
    4. 输出

  5. 测试评估

    (1) 缺陷评估 (2) 覆盖评测 (3) 质量评测

  6. 测试总结

软件测试计划

软件测试计划的作用：

  • 使软件测试工作进行更顺利
  • 增进项目参加人员之间的沟通
  • 及早发现和修正软件规格说明书的问题
  • 使软件测试工作更易于管理

制定测试计划的原则：

  1. 制定测试计划应该尽早开始
  2. 保持测试计划的灵活性
  3. 保持测试计划简洁易读
  4. 尽量争取多方面来评审测试计划
  5. 计算测试计划的投入
